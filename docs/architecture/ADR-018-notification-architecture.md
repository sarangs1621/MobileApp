# ADR-018 — Notification Architecture — M10

**Status:** Proposed — **M10 Step 1 (design)** · **Date:** 2026-07-11 · **Deciders:** Architecture, Product
**Related:** ADR-002 (business layer is the authorization gate) · ADR-003 (repositories; Prisma only in `packages/db`) ·
ADR-005 (`@repo/notifications` — the channel-**delivery** seam: IN_APP/PUSH/SMS/WHATSAPP adapters) · ADR-007 (AuditLog in-transaction) ·
ADR-008 (loose `schoolId`) · ADR-011/012/013 (**recipients derived from `Enrollment`/`TeacherAssignment`**, resolved once at emit time) ·
ADR-017 (Timetable — the publish/mutation surface wrapped for `TIMETABLE_UPDATED`) ·
DATABASE_CONVENTIONS (enums, Restrict, loose `schoolId`, in-tx audit) · PERMISSIONS_MATRIX (`notification:manage_own` self-scope — adopted; `announcement:send` added here) · CODING_STANDARDS §1/§4 (DTOs, layering)
**Precedes:** M10 (Notifications & Communication) — this ADR fixes the design; Steps 2–10 execute it.

---

> **Milestone framing.** This is **M10 — Notifications & Communication**, a new **in-app** notification domain
> over frozen M1–M9: a `Notification` record + explicit per-user `NotificationRecipient` copies, generated by
> business services **after** a publish transaction commits. It is **purely additive** — two new tables, two new
> enums, and a post-commit `emit*` call added to existing **transport** (routers) — **zero change to any frozen
> business service or table** (proven by `prisma migrate diff` at Step 2). **NO chat, push, SMS, WhatsApp or email**
> — those are the existing ADR-005 delivery seam's future job, not M10.

## Context

Frozen M1–M9 already have publish flows — `publishHomework`, `publishExam`, `publishReportCard` (each a one-way
DRAFT→PUBLISHED transition), and the timetable entry CRUD (ADR-017). None of them notify anyone today (verified:
`publishExam` even documents "Sends NO notification"). M10 makes those events visible **inside the app**: a bell,
an unread badge, a list, read/archive tracking, and a deep link to the destination screen.

Four facts already in the codebase constrain the design:

1. **`@repo/notifications` already exists and is the *delivery* seam** (ADR-005) — `NotificationService.send(event)`
   routing a channel-agnostic message to IN_APP/PUSH/SMS/WHATSAPP **adapters**, wired as `ctx.notifications` and
   seeded empty (no adapters) since M1. It is about **outbound delivery to a channel**, not persistence. M10's
   in-app notification is a **different layer**: a **persisted** record users read via tRPC. The `IN_APP` channel
   key already exists (`NOTIFICATION_CHANNELS`). So M10 = *persist the IN_APP notification*; the delivery seam is
   **the reserved future provider seam** — PUSH/SMS later fan out the same event with **zero** change to M10's core.
2. **No `Notification`/`NotificationRecipient`/`Announcement` model exists** — this is a green-field additive domain
   (a `DeviceToken` table exists from M1 for future push registration; out of scope here).
3. **Recipients are derivable from existing services** — `enrollments.listBySection` → `studentParents.listByStudent`
   → parent→user for "parents of a section"; `teacherAssignments`/`examSections` for "teachers of an exam". The brief
   mandates recipients be **stored explicitly** (no dynamic recipient query at read time), so we resolve once at emit
   and persist a `NotificationRecipient` row per user.
4. **House idioms** apply unchanged: enums, `Restrict` on every FK, loose `schoolId`, AuditLog written by the service
   in the same transaction (ADR-007), the M9 service/repository/DTO template.

## Decision

### 1. Two additive models — `Notification` (shared record) + `NotificationRecipient` (per-user copy) — and two enums

```
                       ┌── isRead/readAt        (read tracking)
User ──1:N── NotificationRecipient ─ isArchived/archivedAt  (soft delete)
                        │  Restrict
Notification ──1:N──────┘
  schoolId, type, priority, title, body, actionUrl, createdAt   (immutable)
```

**`Notification`** — one immutable event record: `id, schoolId, type NotificationType, priority NotificationPriority,
title, body, actionUrl String?, createdAt`. **No `updatedAt`** — the event never changes; only per-user state does.
`schoolId` is a **loose** ref (ADR-008, no FK). `actionUrl` is the **deep link** (`/homework/:id`, `/report-cards/:id`,
`/exams/:id`, `/timetable`) the mobile/web client routes on.

**`NotificationRecipient`** — one user's copy of a notification: `id, notificationId, userId, isRead Boolean
@default(false), readAt DateTime?, isArchived Boolean @default(false), archivedAt DateTime?, createdAt`. Both FKs
**Restrict** (`notification → Notification`, `user → User`). **`@@unique([notificationId, userId])`** — a user is
notified once per event (idempotency floor). Indexes per brief: **`[userId, isRead, createdAt]`** (the unread-list /
badge path), **`[notificationId]`**, **`[createdAt]`**.

**`enum NotificationType { HOMEWORK HOMEWORK_PUBLISHED EXAM_PUBLISHED REPORT_CARD_PUBLISHED TIMETABLE_UPDATED
STUDY_MATERIAL ANNOUNCEMENT SYSTEM }`** — the full brief set. `HOMEWORK_PUBLISHED`/`EXAM_PUBLISHED`/
`REPORT_CARD_PUBLISHED`/`ANNOUNCEMENT` are wired in M10; `HOMEWORK` (generic), `STUDY_MATERIAL`, `TIMETABLE_UPDATED`
and `SYSTEM` are **reserved** — declared so future events are additive, with no M10 source (see deviation #1/#2).

**`enum NotificationPriority { LOW NORMAL HIGH URGENT }`** — display/sort weight; NORMAL default.

### 2. `archive` = soft delete (reversible); `delete` = hard-remove the caller's own recipient row

The given field list has **`isArchived`/`archivedAt` but no `isDeleted`**, so:

- **`archive()` / `unarchive()`** flip `isArchived` (+ `archivedAt`) — the reversible **soft delete**; archived rows
  drop out of the default list but survive.
- **`delete()`** hard-deletes **that user's `NotificationRecipient` row only**. The shared `Notification` and every
  other recipient are untouched (the recipient is a leaf; Restrict permits deleting the child). No `isDeleted` column
  is invented. *(Forced by the field list — flagged at STOP, deviation #3.)*

### 3. Generation — a business-layer `*AndNotify` **composition**; the router calls one thin business fn (STEP 5)

The freeze rule is **"Notification delivery wraps those services instead of changing them."** The wrap is a **new
business-layer composition** — orchestration stays in the business layer, **not** in transport — that calls the
frozen publish service, then, once it has committed, fires the emit:

```ts
// packages/business/src/services/notification/publish-with-notify.ts  (BUSINESS layer, additive)
export async function publishHomeworkAndNotify(ctx, homeworkId) {
  const hw = await publishHomework(ctx, homeworkId);  // frozen service — UNCHANGED, commits its own tx + audit
  await emitHomeworkPublished(ctx, hw);               // resolve recipients + createBulk (best-effort)
  return hw;
}

// homework router (transport) stays THIN — one business call, no orchestration:
publish: …mutation(({ ctx, input }) => publishHomeworkAndNotify(createServiceContext(ctx.user), input.homeworkId))
```

- **`emit*` runs only after the publish transaction has committed** (`publishHomework` returned) — satisfies "events
  never fire before commit". Recipients are **resolved once and persisted** (`createBulk` = one `Notification` + N
  `NotificationRecipient` rows in one tx, audited) — no dynamic recipient query at read time.
- **Best-effort, non-blocking:** the publish already succeeded and is committed; an `emit*` failure is **caught and
  logged**, never propagated — a notification-write hiccup must not fail (or appear to fail) a committed publish.
- **No new business logic in the frozen services** — the `emit*` orchestrator and the `*AndNotify` composer both live
  in the **new** notification domain (`packages/business/src/services/notification/`); the composer calls the frozen
  publish service without touching it. The `migrate diff` / DB freeze proof is unaffected; the only change to a frozen
  file is the **router repointing to the composer** (still one thin business call). *(Deviation #4.)*
- **Canonical pattern (recorded for future milestones).** Any later milestone adding notifications to an existing
  action MUST follow this shape: add an `emit*` + a `*AndNotify` composer in the notification domain and repoint the
  router — **never** inline the publish-then-notify orchestration in transport, and **never** edit the frozen service.
  This keeps every notification integration consistent and the transport layer thin (ADR-002).
- **"No duplicate notifications"** = the existing publish guard. `publishHomework`/`publishExam`/`publishReportCard`
  each reject a re-publish of an already-published entity (Conflict thrown before `emit*` is reached), so a client
  retry can't double-fire. A **legitimate** second publish (homework reopen→republish; report-card
  correction/version-bump) **does** emit a new notification — it's a real new event (decision recorded, deviation #5).

### 4. Recipient resolution — concrete rules (reuse Enrollment / TeacherAssignment; no all-school fan-out)

| Event | Recipients | Resolution (reused services) | Type / priority |
|---|---|---|---|
| **Homework publish** | parents of the section | `enrollments.listBySection(year, sectionId)` → `studentParents.listByStudent` → parent→user | `HOMEWORK_PUBLISHED` / NORMAL |
| **Exam publish** | **the exam's** section+subject teachers (**not** all school teachers) | the exam's `ExamSection`s → those sections' `teacherAssignments` | `EXAM_PUBLISHED` / NORMAL |
| **Report card publish** | the card's parent(s) | card `enrollment → student` → `studentParents.listByStudent` → parent→user | `REPORT_CARD_PUBLISHED` / HIGH |
| **Announcement** (admin) | **chosen scope**: `SCHOOL` = every active parent+teacher of the school; `SECTION` = that section's parents + assigned teachers | `people` / `enrollments.listBySection` + `teacherAssignments` | `ANNOUNCEMENT` / by admin (default NORMAL) |
| **Timetable** | teachers + parents of the section | *(deferred — deviation #1)* | `TIMETABLE_UPDATED` / NORMAL |
| **Study material** | parents | *(no source feature — deviation #2)* | `STUDY_MATERIAL` |

Announcement needs **no `Announcement` table** — it is a `Notification(type=ANNOUNCEMENT)` with explicitly-resolved
recipients, created by a new `createAnnouncement` admin action (Step 6).

### 5. RLS (STEP 3) — a user reads only their own recipient rows; admin ALL

Defense-in-depth (the business layer is the real gate; the app reaches these tables as `service_role`, BYPASSRLS).

| Table | Admin | Teacher / Parent | Anon |
|---|---|---|---|
| `NotificationRecipient` | ALL | **SELECT/UPDATE own** (`userId = auth.uid()`) — read state, archive | none |
| `Notification` | ALL | SELECT **iff** a recipient row for `auth.uid()` exists (`EXISTS`) | none |

Isolation proofs (Step 3, rolled back): **Teacher A cannot read Teacher B**, **Parent cannot read another parent**
(both forced by `userId = auth.uid()` on the recipient), **Admin sees all**, **Anon none**.

### 6. Permissions — self-scoped recipient ops + an admin announcement grant (permission-only, no flag)

- **`notification:manage_own`** (`NOTIFICATION_MANAGE_OWN`) — list, unreadCount, markRead, markAllRead, archive,
  unarchive, delete; **self-scope for every role** (the matrix already lists it). Row scope (`userId = self`) is
  enforced in the service.
- **`announcement:send`** (`ANNOUNCEMENT_SEND`) — SA/OA only, for `createAnnouncement`.
- **No feature flag** — notifications are core (the ADR-013/M6, ADR-017/M9 precedent; no flag infra exists).

### 7. Future provider seam (recorded)

M10 persists the **IN_APP** notification only. PUSH/SMS/WhatsApp are added later by registering adapters on the
existing `@repo/notifications` `NotificationService` (ADR-005) and having the same `emit*` orchestrator also call
`ctx.notifications.send(event)` — **additive**, no change to M10's persistence, DTOs, RLS or API. `DeviceToken`
(M1) is the future push-registration table. **Notification preferences** (opt-out per type/channel) are explicitly
out of scope — a future additive table keyed on `(userId, type)`.

## Deviations from the literal brief (flagged for veto at STOP)

1. **Timetable notifications — RECOMMEND DEFER (do not auto-emit per entry).** Timetable has **no publish
   transition**; entries mutate one at a time (`createEntry/updateEntry/deleteEntry`), and the Step-8 console saves
   **one cell per modal**. Auto-emitting `TIMETABLE_UPDATED` per entry means building a ~40-slot grid fires ~40 ×
   (teachers+parents) notifications — a storm that also cuts against the "no duplicate notifications" DoD. There is
   no coarse "timetable published" action to hook. **Recommendation:** ship the `TIMETABLE_UPDATED` type + recipient
   rule but **defer automatic emission** to a future coarse "publish/notify timetable" action. Alternatives for the
   STOP: **(a)** emit per-entry with a documented `// ponytail:` storm ceiling; **(b)** *[recommended]* defer;
   **(c)** add a manual admin "notify section of timetable change" action in M10 (reuses the announcement path).
2. **No `STUDY_MATERIAL` source in M10.** There is **no study-material feature** in the repo — nothing to wrap. The
   enum value is **reserved** (future-additive); M10 wires no study-material event.
3. **`delete()` = hard-delete the caller's recipient row** (no `isDeleted` field given). Archive is the soft delete
   (decision #2). Only reading consistent with the Step-2 field list.
4. **The wrap is a business-layer `*AndNotify` composition; each frozen publish router repoints to it.** Business
   **services** and all tables stay byte-identical (the composer *calls* the frozen service, never edits it); the only
   change to a frozen file is the router swapping `publishX` → `publishXAndNotify` — one thin business call, no
   orchestration in transport. The freeze proof is `migrate diff` (DB-only) + "no service file changed". *(Revised per
   Step-5 review: orchestration moved out of transport into the business layer and recorded as the canonical pattern —
   §3.)*
5. **A legitimate re-publish emits again** (homework reopen→republish; report-card correction). Allowed — content
   changed, it's a new event. Client-retry double-fire is prevented by the existing publish guard, not by us.

## Alternatives considered

1. **Emit from inside the existing publish services** (an event hook in `publishHomework`). **Rejected** — modifies
   frozen business logic, the exact thing the freeze rule forbids. The post-commit transport wrap is additive.
2. **A generic event bus / outbox table** subscribed by the notification domain. **Rejected for M10** — speculative
   (YAGNI); no async worker exists, delivery is synchronous per the brief. The direct post-commit `emit*` is the lazy
   correct seam; an outbox is an additive migration if a real async requirement ever lands.
3. **Reuse `@repo/notifications` (`ctx.notifications.send`) as the persistence path.** **Rejected** — that package is
   the *channel-delivery* abstraction (outbound to a provider), not a store. M10 needs a queryable, read-trackable
   record. They compose (decision #7), they don't merge.
4. **A dynamic recipient query at read time** (compute "who should see this" when the list is fetched). **Rejected** —
   the brief mandates explicit stored recipients; dynamic resolution can't do per-user read/archive state and would
   re-run enrollment/assignment joins on every list. Resolve-once-and-persist is both required and cheaper.
5. **A separate `Announcement` table.** **Rejected** — an announcement is just a `Notification(type=ANNOUNCEMENT)`
   with resolved recipients; a table adds nothing M10 needs.
6. **Soft-delete `delete()` via a new `isDeleted` column.** **Rejected** — not in the given field list; archive
   already provides reversible hiding. Hard-deleting the leaf recipient row is the minimal correct semantics.

## Consequences

- (+) **Purely additive** — two tables + two enums; every frozen **business service and table untouched** (proven by
  `migrate diff` at Step 2 + "no service file changed").
- (+) **Events fire after commit, once, with explicit stored recipients** — read/archive state is per-user and cheap
  to query (`[userId, isRead, createdAt]`).
- (+) **Clean future seam** — PUSH/SMS fan out from the same `emit*` via the existing ADR-005 delivery abstraction
  with no change to M10's core; preferences are a future keyed table.
- (+) **No new RLS *shape*** — the standard admin-ALL / own-row / anon-none idiom.
- (−) **Transport touched** — one post-commit `emit*` line per publish router (deviation #4).
- (−) **Timetable + study-material notifications not wired in M10** (deviations #1/#2) — types reserved, sources
  deferred.
- (−) **Best-effort emit** — a notification-write failure after a committed publish is logged, not surfaced; the
  event is lost (no retry). Acceptable for in-app; the future outbox (alternative #2) is the durability upgrade path.

## STOP — Step 1 boundary

This ADR fixes the design. **No migration, SQL, DTO, or UI is written here** — Step 2 (additive migration +
`migrate diff` proof) onward executes it. Five items need an explicit nod before Step 2, because they shape the
schema/scope: **(a)** timetable notifications = **defer** auto-emit (deviation #1 — the one real design call);
**(b)** `delete()` = hard-delete own recipient row, archive = soft delete (#3); **(c)** the wrap is a business-layer
`*AndNotify` composition the routers repoint to (#4 — revised: orchestration in business, not transport); **(d)**
exam→section/subject teachers, announcement→SCHOOL|SECTION scope
(decision #4); **(e)** re-publish emits again (#5).
