---

> **Milestone framing.** This is **M10 ‚Äî Notifications & Communication**, a new **in-app** notification domain
> over frozen M1‚ÄìM9: a `Notification` record + explicit per-user `NotificationRecipient` copies, generated by
> business services **after** a publish transaction commits. It is **purely additive** ‚Äî two new tables, two new
> enums, and a post-commit `emit*` call added to existing **transport** (routers) ‚Äî **zero change to any frozen
> business service or table** (proven by `prisma migrate diff` at Step 2). **NO chat, push, SMS, WhatsApp or email**
> ‚Äî those are the existing ADR-005 delivery seam's future job, not M10.

## Context

Frozen M1‚ÄìM9 already have publish flows ‚Äî `publishHomework`, `publishExam`, `publishReportCard` (each a one-way
DRAFT‚ÜíPUBLISHED transition), and the timetable entry CRUD (ADR-017). None of them notify anyone today (verified:
`publishExam` even documents "Sends NO notification"). M10 makes those events visible **inside the app**: a bell,
an unread badge, a list, read/archive tracking, and a deep link to the destination screen.

Four facts already in the codebase constrain the design:

1. **`@repo/notifications` already exists and is the *delivery* seam** (ADR-005) ‚Äî `NotificationService.send(event)`
   routing a channel-agnostic message to IN_APP/PUSH/SMS/WHATSAPP **adapters**, wired as `ctx.notifications` and
   seeded empty (no adapters) since M1. It is about **outbound delivery to a channel**, not persistence. M10's
   in-app notification is a **different layer**: a **persisted** record users read via tRPC. The `IN_APP` channel
   key already exists (`NOTIFICATION_CHANNELS`). So M10 = *persist the IN_APP notification*; the delivery seam is
   **the reserved future provider seam** ‚Äî PUSH/SMS later fan out the same event with **zero** change to M10's core.
2. **No `Notification`/`NotificationRecipient`/`Announcement` model exists** ‚Äî this is a green-field additive domain
   (a `DeviceToken` table exists from M1 for future push registration; out of scope here).
3. **Recipients are derivable from existing services** ‚Äî `enrollments.listBySection` ‚Üí `studentParents.listByStudent`
   ‚Üí parent‚Üíuser for "parents of a section"; `teacherAssignments`/`examSections` for "teachers of an exam". The brief
   mandates recipients be **stored explicitly** (no dynamic recipient query at read time), so we resolve once at emit
   and persist a `NotificationRecipient` row per user.
4. **House idioms** apply unchanged: enums, `Restrict` on every FK, loose `schoolId`, AuditLog written by the service
   in the same transaction (ADR-007), the M9 service/repository/DTO template.

## Decision

### 1. Two additive models ‚Äî `Notification` (shared record) + `NotificationRecipient` (per-user copy) ‚Äî and two enums

```
                       ‚îå‚îÄ‚îÄ isRead/readAt        (read tracking)
User ‚îÄ‚îÄ1:N‚îÄ‚îÄ NotificationRecipient ‚îÄ isArchived/archivedAt  (soft delete)
                        ‚îÇ  Restrict
Notification ‚îÄ‚îÄ1:N‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò
  schoolId, type, priority, title, body, actionUrl, createdAt   (immutable)
```

**`Notification`** ‚Äî one immutable event record: `id, schoolId, type NotificationType, priority NotificationPriority,
title, body, actionUrl String?, createdAt`. **No `updatedAt`** ‚Äî the event never changes; only per-user state does.
`schoolId` is a **loose** ref (ADR-008, no FK). `actionUrl` is the **deep link** (`/homework/:id`, `/report-cards/:id`,
`/exams/:id`, `/timetable`) the mobile/web client routes on.

**`NotificationRecipient`** ‚Äî one user's copy of a notification: `id, notificationId, userId, isRead Boolean
@default(false), readAt DateTime?, isArchived Boolean @default(false), archivedAt DateTime?, createdAt`. Both FKs
**Restrict** (`notification ‚Üí Notification`, `user ‚Üí User`). **`@@unique([notificationId, userId])`** ‚Äî a user is
notified once per event (idempotency floor). Indexes per brief: **`[userId, isRead, createdAt]`** (the unread-list /
badge path), **`[notificationId]`**, **`[createdAt]`**.

**`enum NotificationType { HOMEWORK HOMEWORK_PUBLISHED EXAM_PUBLISHED REPORT_CARD_PUBLISHED TIMETABLE_UPDATED
STUDY_MATERIAL ANNOUNCEMENT SYSTEM }`** ‚Äî the full brief set. `HOMEWORK_PUBLISHED`/`EXAM_PUBLISHED`/
`REPORT_CARD_PUBLISHED`/`ANNOUNCEMENT` are wired in M10; `HOMEWORK` (generic), `STUDY_MATERIAL`, `TIMETABLE_UPDATED`
and `SYSTEM` are **reserved** ‚Äî declared so future events are additive, with no M10 source (see deviation #1/#2).

**`enum NotificationPriority { LOW NORMAL HIGH URGENT }`** ‚Äî display/sort weight; NORMAL default.

### 2. `archive` = soft delete (reversible); `delete` = hard-remove the caller's own recipient row

The given field list has **`isArchived`/`archivedAt` but no `isDeleted`**, so:

- **`archive()` / `unarchive()`** flip `isArchived` (+ `archivedAt`) ‚Äî the reversible **soft delete**; archived rows
  drop out of the default list but survive.
- **`delete()`** hard-deletes **that user's `NotificationRecipient` row only**. The shared `Notification` and every
  other recipient are untouched (the recipient is a leaf; Restrict permits deleting the child). No `isDeleted` column
  is invented. *(Forced by the field list ‚Äî flagged at STOP, deviation #3.)*

### 3. Generation ‚Äî post-commit `emit*` from **transport**, business services untouched (STEP 5)

The freeze rule is **"Notification delivery wraps those services instead of changing them."** The wrap seam is the
**router** (thin transport, not business logic):

```ts
// homework router (transport) ‚Äî the ONLY change to a frozen file: one post-commit line
const hw = await publishHomework(ctx, input);   // frozen service ‚Äî UNCHANGED, commits its own tx + audit
await emitHomeworkPublished(ctx, hw);           // NEW business orchestrator ‚Äî resolve recipients + createBulk
return hw;
```

- **`emit*` runs only after the publish transaction has committed** (`publishHomework` returned) ‚Äî satisfies "events
  never fire before commit". Recipients are **resolved once and persisted** (`createBulk` = one `Notification` + N
  `NotificationRecipient` rows in one tx, audited) ‚Äî no dynamic recipient query at read time.
- **Best-effort, non-blocking:** the publish already succeeded and is committed; an `emit*` failure is **caught and
  logged**, never propagated ‚Äî a notification-write hiccup must not fail (or appear to fail) a committed publish.
- **No new business logic in the frozen services** ‚Äî `emit*` orchestrators live in the **new** notification domain
  (`packages/business/src/services/notification/`). The `migrate diff` / DB freeze proof is unaffected; the code
  change is **one post-commit call per publish router** (M6/M7 + announcement). *(Stated for knowing veto ‚Äî
  deviation #4.)*
- **"No duplicate notifications"** = the existing publish guard. `publishHomework`/`publishExam`/`publishReportCard`
  each reject a re-publish of an already-published entity (Conflict thrown before `emit*` is reached), so a client
  retry can't double-fire. A **legitimate** second publish (homework reopen‚Üírepublish; report-card
  correction/version-bump) **does** emit a new notification ‚Äî it's a real new event (decision recorded, deviation #5).

### 4. Recipient resolution ‚Äî concrete rules (reuse Enrollment / TeacherAssignment; no all-school fan-out)

| Event | Recipients | Resolution (reused services) | Type / priority |
|---|---|---|---|
| **Homework publish** | parents of the section | `enrollments.listBySection(year, sectionId)` ‚Üí `studentParents.listByStudent` ‚Üí parent‚Üíuser | `HOMEWORK_PUBLISHED` / NORMAL |
| **Exam publish** | **the exam's** section+subject teachers (**not** all school teachers) | the exam's `ExamSection`s ‚Üí those sections' `teacherAssignments` | `EXAM_PUBLISHED` / NORMAL |
| **Report card publish** | the card's parent(s) | card `enrollment ‚Üí student` ‚Üí `studentParents.listByStudent` ‚Üí parent‚Üíuser | `REPORT_CARD_PUBLISHED` / HIGH |
| **Announcement** (admin) | **chosen scope**: `SCHOOL` = every active parent+teacher of the school; `SECTION` = that section's parents + assigned teachers | `people` / `enrollments.listBySection` + `teacherAssignments` | `ANNOUNCEMENT` / by admin (default NORMAL) |
| **Timetable** | teachers + parents of the section | *(deferred ‚Äî deviation #1)* | `TIMETABLE_UPDATED` / NORMAL |
| **Study material** | parents | *(no source feature ‚Äî deviation #2)* | `STUDY_MATERIAL` |

Announcement needs **no `Announcement` table** ‚Äî it is a `Notification(type=ANNOUNCEMENT)` with explicitly-resolved
recipients, created by a new `createAnnouncement` admin action (Step 6).

### 5. RLS (STEP 3) ‚Äî a user reads only their own recipient rows; admin ALL

Defense-in-depth (the business layer is the real gate; the app reaches these tables as `service_role`, BYPASSRLS).

| Table | Admin | Teacher / Parent | Anon |
|---|---|---|---|
| `NotificationRecipient` | ALL | **SELECT/UPDATE own** (`userId = auth.uid()`) ‚Äî read state, archive | none |
| `Notification` | ALL | SELECT **iff** a recipient row for `auth.uid()` exists (`EXISTS`) | none |

Isolation proofs (Step 3, rolled back): **Teacher A cannot read Teacher B**, **Parent cannot read another parent**
(both forced by `userId = auth.uid()` on the recipient), **Admin sees all**, **Anon none**.

### 6. Permissions ‚Äî self-scoped recipient ops + an admin announcement grant (permission-only, no flag)

- **`notification:manage_own`** (`NOTIFICATION_MANAGE_OWN`) ‚Äî list, unreadCount, markRead, markAllRead, archive,
  unarchive, delete; **self-scope for every role** (the matrix already lists it). Row scope (`userId = self`) is
  enforced in the service.
- **`announcement:send`** (`ANNOUNCEMENT_SEND`) ‚Äî SA/OA only, for `createAnnouncement`.
- **No feature flag** ‚Äî notifications are core (the ADR-013/M6, ADR-017/M9 precedent; no flag infra exists).

### 7. Future provider seam (recorded)

M10 persists the **IN_APP** notification only. PUSH/SMS/WhatsApp are added later by registering adapters on the
existing `@repo/notifications` `NotificationService` (ADR-005) and having the same `emit*` orchestrator also call
`ctx.notifications.send(event)` ‚Äî **additive**, no change to M10's persistence, DTOs, RLS or API. `DeviceToken`
(M1) is the future push-registration table. **Notification preferences** (opt-out per type/channel) are explicitly
out of scope ‚Äî a future additive table keyed on `(userId, type)`.

## Deviations from the literal brief (flagged for veto at STOP)

1. **Timetable notifications ‚Äî RECOMMEND DEFER (do not auto-emit per entry).** Timetable has **no publish
   transition**; entries mutate one at a time (`createEntry/updateEntry/deleteEntry`), and the Step-8 console saves
   **one cell per modal**. Auto-emitting `TIMETABLE_UPDATED` per entry means building a ~40-slot grid fires ~40 √ó
   (teachers+parents) notifications ‚Äî a storm that also cuts against the "no duplicate notifications" DoD. There is
   no coarse "timetable published" action to hook. **Recommendation:** ship the `TIMETABLE_UPDATED` type + recipient
   rule but **defer automatic emission** to a future coarse "publish/notify timetable" action. Alternatives for the
   STOP: **(a)** emit per-entry with a documented `// ponytail:` storm ceiling; **(b)** *[recommended]* defer;
   **(c)** add a manual admin "notify section of timetable change" action in M10 (reuses the announcement path).
2. **No `STUDY_MATERIAL` source in M10.** There is **no study-material feature** in the repo ‚Äî nothing to wrap. The
   enum value is **reserved** (future-additive); M10 wires no study-material event.
3. **`delete()` = hard-delete the caller's recipient row** (no `isDeleted` field given). Archive is the soft delete
   (decision #2). Only reading consistent with the Step-2 field list.
4. **The wrap adds one post-commit `emit*` call to each frozen publish router (M6/M7).** Business **services** and
   all tables stay byte-identical; the freeze proof is `migrate diff` (DB-only) + "no service file changed". Called
   out so the transport edit is vetoed knowingly.
5. **A legitimate re-publish emits again** (homework reopen‚Üírepublish; report-card correction). Allowed ‚Äî content
   changed, it's a new event. Client-retry double-fire is prevented by the existing publish guard, not by us.

## Alternatives considered

1. **Emit from inside the existing publish services** (an event hook in `publishHomework`). **Rejected** ‚Äî modifies
   frozen business logic, the exact thing the freeze rule forbids. The post-commit transport wrap is additive.
2. **A generic event bus / outbox table** subscribed by the notification domain. **Rejected for M10** ‚Äî speculative
   (YAGNI); no async worker exists, delivery is synchronous per the brief. The direct post-commit `emit*` is the lazy
   correct seam; an outbox is an additive migration if a real async requirement ever lands.
3. **Reuse `@repo/notifications` (`ctx.notifications.send`) as the persistence path.** **Rejected** ‚Äî that package is
   the *channel-delivery* abstraction (outbound to a provider), not a store. M10 needs a queryable, read-trackable
   record. They compose (decision #7), they don't merge.
4. **A dynamic recipient query at read time** (compute "who should see this" when the list is fetched). **Rejected** ‚Äî
   the brief mandates explicit stored recipients; dynamic resolution can't do per-user read/archive state and would
   re-run enrollment/assignment joins on every list. Resolve-once-and-persist is both required and cheaper.
5. **A separate `Announcement` table.** **Rejected** ‚Äî an announcement is just a `Notification(type=ANNOUNCEMENT)`
   with resolved recipients; a table adds nothing M10 needs.
6. **Soft-delete `delete()` via a new `isDeleted` column.** **Rejected** ‚Äî not in the given field list; archive
   already provides reversible hiding. Hard-deleting the leaf recipient row is the minimal correct semantics.

## Consequences

- (+) **Purely additive** ‚Äî two tables + two enums; every frozen **business service and table untouched** (proven by
  `migrate diff` at Step 2 + "no service file changed").
- (+) **Events fire after commit, once, with explicit stored recipients** ‚Äî read/archive state is per-user and cheap
  to query (`[userId, isRead, createdAt]`).
- (+) **Clean future seam** ‚Äî PUSH/SMS fan out from the same `emit*` via the existing ADR-005 delivery abstraction
  with no change to M10's core; preferences are a future keyed table.
- (+) **No new RLS *shape*** ‚Äî the standard admin-ALL / own-row / anon-none idiom.
- (‚àí) **Transport touched** ‚Äî one post-commit `emit*` line per publish router (deviation #4).
- (‚àí) **Timetable + study-material notifications not wired in M10** (deviations #1/#2) ‚Äî types reserved, sources
  deferred.
- (‚àí) **Best-effort emit** ‚Äî a notification-write failure after a committed publish is logged, not surfaced; the
  event is lost (no retry). Acceptable for in-app; the future outbox (alternative #2) is the durability upgrade path.

## STOP ‚Äî Step 1 boundary

This ADR fixes the design. **No migration, SQL, DTO, or UI is written here** ‚Äî Step 2 (additive migration +
`migrate diff` proof) onward executes it. Five items need an explicit nod before Step 2, because they shape the
schema/scope: **(a)** timetable notifications = **defer** auto-emit (deviation #1 ‚Äî the one real design call);
**(b)** `delete()` = hard-delete own recipient row, archive = soft delete (#3); **(c)** the wrap adds a post-commit
`emit*` to frozen publish **routers** (#4); **(d)** exam‚Üísection/subject teachers, announcement‚ÜíSCHOOL|SECTION scope
(decision #4); **(e)** re-publish emits again (#5).
