# Feature — Notifications & Communication (M10)

**Spec:** `docs/architecture/ADR-018-notification-architecture.md` · `docs/milestones/M10.md`
**Status:** Implemented (M10) — awaiting milestone approval.

In-app notifications over frozen M1–M9: a `Notification` event record + explicit per-user
`NotificationRecipient` copies, generated by business services **after** a publish transaction commits.
A bell + unread badge + inbox on mobile and web, read/archive tracking, deep links, and an admin
announcement composer. **No push / SMS / WhatsApp / email / chat** — those are the ADR-005 delivery
seam's future job.

## Model (grain)

```
                       ┌── isRead/readAt        (read tracking)
User ──1:N── NotificationRecipient ─ isArchived/archivedAt  (soft delete)
                        │  Restrict
Notification ──1:N──────┘
  schoolId, type, priority, title, body, actionUrl, createdAt   (immutable event)
```

- **Notification** — one immutable event (no `updatedAt`); `schoolId` loose (ADR-008); `actionUrl` = deep link.
- **NotificationRecipient** — one user's copy: `isRead`/`readAt` (read tracking), `isArchived`/`archivedAt`
  (reversible **soft delete**). `@@unique(notificationId, userId)` = one copy per user per event.
- Both FKs **Restrict**. Indexes: `(userId, isRead, createdAt)`, `(notificationId)`, `(createdAt)`.
- `enum NotificationType` (HOMEWORK · HOMEWORK_PUBLISHED · EXAM_PUBLISHED · REPORT_CARD_PUBLISHED ·
  TIMETABLE_UPDATED · STUDY_MATERIAL · ANNOUNCEMENT · SYSTEM) · `enum NotificationPriority` (LOW..URGENT).

## Generation (event integration)

Notifications are generated by a **business-layer `*AndNotify` composition** — never inside the frozen publish
service, never in transport (ADR-018 §3, the canonical pattern). The router calls one thin business fn:

```ts
publishReportCardAndNotify(ctx, id) = publishReportCard(ctx, id)  // frozen — untouched, commits its own tx
                                    → emitReportCardPublished(ctx, card)  // post-commit, best-effort
```

- Events fire **after commit** (never before); a re-publish (reopen, correction) emits again by design.
- **Best-effort** — an emit failure is logged, never propagated (a committed publish never fails on a notify hiccup).
- **No duplicates** — the existing publish guard rejects a re-publish of an already-published entity, so a client
  retry can't double-fire.

| Event | Recipients | Type / priority |
|---|---|---|
| Homework publish | parents of the section (`enrollments → studentParents → parents`) | HOMEWORK_PUBLISHED / NORMAL |
| Exam publish | teachers of the exam's sections (`assessments → examSections → teacherAssignments`) | EXAM_PUBLISHED / NORMAL |
| Report card publish | the card's parent(s) | REPORT_CARD_PUBLISHED / HIGH |
| Announcement (admin) | SCHOOL (active parents+teachers) or SECTION (that section) | ANNOUNCEMENT / admin-chosen |

Recipients are **resolved once at emit and stored explicitly** (reuse Enrollment / TeacherAssignment — no
dynamic recipient query at read time). Parents without a login user are skipped. **Deferred** (ADR-018
deviations): timetable auto-emit (no publish transition → per-entry would storm; `TIMETABLE_UPDATED` reserved)
and study material (no source feature).

## Surface

- **Business:** `packages/business/src/services/notification/` — `notification.service` (create/createBulk,
  list, unreadCount, markRead, markAllRead, archive/unarchive, delete — self-scope, every mutation audited),
  `recipients`, `events` (emit*), `publish-with-notify` (`*AndNotify`), `announcement.service`.
- **API:** `notification.*` — `list`, `unreadCount`, `markRead`, `markAllRead`, `archive`, `unarchive`,
  `delete` (all `notification:manage_own`), `createAnnouncement` (`announcement:send`).
- **Mobile:** home-header bell + badge; `/notifications` inbox (FlatList + pull-to-refresh, tap = mark read +
  deep-link, per-row archive, mark-all-read).
- **Web:** dashboard-header bell + badge + dropdown; `/notifications` page (inbox + admin announcement composer —
  bulk school-wide or one section).

## Authorization

Permission-only (no feature flag). `notification:manage_own` → **every role** (self-scope: `userId = self`,
enforced in the service). `announcement:send` → SUPER_ADMIN + OFFICE_ADMIN. RLS is defense-in-depth: admin ALL;
a user reads/updates/deletes only own recipient rows; a Notification is readable iff an `EXISTS` recipient row;
anon none. Isolation proven — Teacher A ≠ Teacher B, parent ≠ other parent.

## Out of scope

Push (FCM/APNs), SMS, WhatsApp, email, chat/replies, typing/presence, attachments, notification preferences
(opt-out per type/channel) — future milestones. Timetable + study-material notification sources deferred.
