# M3 — People Management — ✅ STEPS 1–10 COMPLETE (awaiting approval)

Scope (per M3 kickoff brief 2026-07-05): **Students (identity), Parents +
StudentParent links, Teacher (Staff) profiles, Enrollment (ADR-010), Student
Documents.** Out: attendance, homework, exams/marks, report cards, fees,
timetable, communication, bulk import — later milestones.

Roles: SUPER_ADMIN + OFFICE_ADMIN full management; TEACHER reads students
enrolled in sections they teach (+ own staff profile, PHOTO documents only);
PARENT reads only their own children (+ own parent record); ACCOUNTANT none.
**ADR-010 is the source of truth for enrollment**: Student stays identity-only;
Enrollment owns (year, class, section, rollNo) with one row per student per
year; promotion creates a new row; history is never mutated.

## Steps

- ✅ 1 — Requirements analysis: lifecycles (student vs enrollment status as
  separate axes), admission/transfer/withdrawal/promotion workflows, edge cases;
  why Enrollment owns placement and Student stays identity-only; future deps
  (attendance/marks/fees hang off Enrollment). Recorded in **ADR-010** +
  PERMISSIONS_MATRIX update. Decision: admission = identity create + separate
  first `enrollment.create` (ADMITTED-before-placement, matches the web UI).
- ✅ 2 — Database design (migration `20260705030000_people_management`): 6
  models (`Student`, `StudentDocument`, `Parent`, `StudentParent`, `Staff`,
  `Enrollment`) + 6 enums. Invariants in the DB: `(schoolId, admissionNo)`
  unique; Aadhaar partial-unique per school; `(studentId, academicYearId)`
  unique; rollNo partial-unique per (year, section) + `CHECK` rollNo-needs-
  section; one-primary-contact partial-unique; `Staff.userId`/`Parent.userId`
  unique. cuid PKs, `@db.Date` calendar columns, loose `schoolId` tenant refs,
  roster/search indexes.
- ✅ 3 — Relationships: Enrollment FKs **Restrict** (student/year/class/
  section — placement history blocks hard deletes; archive instead); documents
  and StudentParent links **Cascade** from Student; `Parent.userId` **SetNull**
  (login removal keeps the contact); `Staff.userId` Cascade with the User.
- ✅ 4 — RLS (migration `20260705040000_people_rls`): RLS ENABLEd on all 5
  tables; admin ALL policies; teacher own-section SELECT (Student/Enrollment/
  StudentDocument via TeacherAssignment); parent own-child SELECT (via
  StudentParent); parent/staff self-SELECT. Defense-in-depth only — the
  business layer is the gate (ADR-002).
- ✅ 5 — Business layer: 5 services (`student`, `enrollment`, `parent`,
  `staff`, `student-document`) + shared `scope.ts` (accessibleStudentIds,
  teacherSectionIds, parentChildIds, recordAudit) + mappers. Permission →
  scope → rule checks, typed DomainErrors, every mutation audited in-tx.
  Teacher document visibility = `TEACHER_VISIBLE_TYPES` (PHOTO only), single
  source of truth.
- ✅ 6 — API layer (`9fded51`): 5 thin routers (`student/parent/teacherProfile/
  enrollment/studentDocument`) on `protectedProcedure`; Zod inputs in
  `@repo/validation`; no logic/Prisma/role strings in transport. Typing-only
  widening of `StudentListFilter` for `exactOptionalPropertyTypes`.
- ✅ 7 — Mobile (`e5b7d28`): read-only screens — students list, student profile
  (identity + ADR-010 enrollment history + guardians), parents, teacher
  profiles; Home links gated per permission via `can()`. Name lookups run only
  for roles holding the matching read permission (fallback: raw ids).
- ✅ 8 — Web (`6f17532`): `/people/*` full CRUD — students (search/status
  filter/create/edit/archive), student detail with enrollment lifecycle
  (enroll/transfer/promote/withdraw modals honoring ADR-010), guardian
  link/unlink (single primary), documents (upload/replace/view/delete);
  parents + teacher-profiles CRUD; permission-filtered tabs; dashboard link.
  **ADR-004 storage implemented end-to-end**: business `StoragePort` +
  `mintDocumentUploadUrl/DownloadUrl` (authz BEFORE any URL; server-chosen
  `schoolId/…` paths), API `storageProcedure` + `uploadUrl/downloadUrl`,
  web service-role adapter (`src/lib/storage.ts`), browser `uploadToSignedUrl`.
  Private `student-documents` bucket documented (runbook §3b) — **manual
  provisioning pending**.
- ✅ 9 — Testing (`d1929eb`): +53 business (identity uniqueness incl. duplicate
  admission no/Aadhaar, archive lifecycle, teacher/parent row scopes incl.
