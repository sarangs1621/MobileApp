# M2 — Academic Foundation — ✅ STEPS 1–10 COMPLETE (awaiting approval)

Scope (supersedes the earlier draft; per M2 kickoff brief 2026-07-05): **academic
structure only** — `AcademicYear`, `AcademicTerm`, `Class`, `Section`, `Subject`,
`TeacherAssignment`. Out: students/enrollment/attendance/marks/report cards/
timetable/fees, people records (Staff/Guardian), bulk import, class-teacher flag.
Naming note: this milestone uses **Class/Section** (older docs said ClassLevel/Division).

Roles: SUPER_ADMIN + OFFICE_ADMIN manage (`academic:manage`); staff read
(`academic:read`); teachers additionally read **their own** assignments via a
`ScopeRule`; parents have no access.

## Steps
- ✅ 1 — Requirements analysis (hierarchy, relationships, future deps, validation
  rules, lifecycles, edge cases, ordering rationale — delivered 2026-07-05).
  Key decisions: Class/Section/Subject are structural (not year-bound; enrollment
  binds time in M3); Subject is a school-wide catalog; `TeacherAssignment.teacherId
  → User.id` (no Staff table), role/status checked in business; assignments NOT
  year-scoped (unique `(teacher, subject, section)` per brief — year scoping is an
  additive M3+ migration if needed); no soft deletes (Restrict FKs + audit rows);
  one-ACTIVE-year via partial unique index; term overlap = inclusive-date
  intersection, enforced in business (DB EXCLUDE constraint decided at Step 2);
  every mutation audited in-transaction.
- ✅ 2 — Database design: 6 models + `AcademicYearStatus` enum in `schema.prisma`;
  migration `20260705000000_academic_structure` applied to the live DB. DB-level
  invariants (all raw SQL, behavior-probed live): partial unique
  `AcademicYear(schoolId) WHERE status='ACTIVE'` (P2002 on 2nd ACTIVE ✓),
  `EXCLUDE USING gist` term overlap incl. boundary touch (✓), `CHECK start<end`
  on year+term (✓), year→term Cascade, all other FKs Restrict. cuid PKs;
  `@db.Date` calendar columns; uniques: `(schoolId,name)` year/class/subject,
  `(yearId,name)` term, `(classId,name)` section, `(teacherId,subjectId,sectionId)`
  assignment. Query indexes: `(schoolId,status)`, `(yearId,startDate)`,
  `(schoolId,sortOrder)`, assignment `sectionId`/`subjectId`/`schoolId` (+teacher
  via unique prefix). `Class.sortOrder` added for deterministic grade ordering.
  User gained a virtual back-relation only (no column/behavior change to frozen M1).
  Audit-compatible: loose `schoolId` scalar everywhere; mutations write `AuditLog`
  from services (Step 4). No student tables.
- ✅ 3 — RLS planning + policies (migration `20260705010000_academic_rls`): RLS
  `ENABLE`d (not FORCE — service_role bypasses) on all 6 tables; `is_academic_admin()`

  + `teacher_read_own_assignments` (SELECT, `teacherId=auth.uid()`) on TeacherAssignment;
  parents/anon deny-by-default. Defense-in-depth only (Prisma bypasses; business layer
  is the gate). Documented in `docs/RLS_POLICIES.md`. Structurally verified (`prisma
  validate` ✓); live `\dp` grant/User-exposure check pending DB creds. Flagged M1 `User`
  no-RLS PII risk for user decision.
- ✅ 4 — Business layer: 6 services (year/term/class/section/subject/teacher-assignment)
