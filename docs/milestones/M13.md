# M13 — Fees & Payment Management — ✅ STEPS 1–9 COMPLETE (awaiting approval)

Scope: a **school fee system** for Indian schools over frozen M1–M12 — fee **structures** (named, per-year templates +
component lines), invoice **generation** (bulk, per section, idempotent, snapshotted total), **partial payments**,
**receipts**, a **parent fee portal**, and M10 notifications. The **first money domain** — all money is **`Int` paise**.
Purely **additive**: 4 new tables (`FeeStructure`, `FeeComponent`, `Invoice`, `Payment`), 2 enums (`InvoiceStatus`,
`PaymentMethod`), 2 new `NotificationType` values, 4 permissions. **NO GST / payroll / accounting / Tally / online
payment gateway / refund workflow / scholarship engine / hostel / transport / inventory / library.** **ADR-021** governs.

> **No online gateway (out of scope) ⇒ parents view, the office records.** v1 parents see dues + view/download receipts;
> payment **recording** is admin office-side (cash/UPI/cheque/…). Refunds + concessions are **deferred** (the model does
> not preclude them — `Payment` is append-only, `FeeComponent` supports a future negative line).

## Deviations (ADR-021)

1. **Refunds deferred** — intro lists them but OUT OF SCOPE excludes "Refund workflow" (refunds "additive later").
   `Payment` is append-only; a refund is a future negative/`REFUND` row.
2. **Concessions deferred** — "Scholarship engine" out of scope; a concession is a future negative `FeeComponent`.
3. **Parents view, admins record** — "Parents can pay" is impossible without the (out-of-scope) online gateway; v1
   parent = view dues + receipts, recording is admin office-side. (Step-1 approval.)
4. **Two new `NotificationType` values** (`INVOICE_ISSUED`, `PAYMENT_RECEIVED`) — an additive `ALTER TYPE … ADD VALUE`,
   not a frozen-table ALTER. (Step-1 approval.)
5. **`Invoice` keeps both `studentId` + `enrollmentId`** — the justified ADR-020 divergence (student ledger needs the
   cross-year person view). (Step-1 approval.)
6. **`OVERDUE` is compute-on-read**, never a stored/transitioned status — no cron infra; the codebase's compute-on-read
   posture.
7. **`generateInvoices` is section-scoped** — the frozen enrollment repo is untouched; class/year coverage iterates
   sections in the web console.
8. **Notifications emitted inline** (not a named `*AndNotify` composer) — these are new M13 actions, not frozen wraps.
